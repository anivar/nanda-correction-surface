▶ New here? What am I looking at?

NANDA is a verifiable directory for AI agents — think DNS, but every answer is signed and the client checks each one itself, so no hop is ever acted on unverified. (Exactly one key is trusted up front: the index’s signing key, which the client pins.)

How to read it: pick a scenario on the left, then use the player to step through. Watch the arrows between the actors in the diagram and read the matching step below — each has a plain-English line and the exact mechanism underneath. A step’s coloured left border matches the diagram’s lines: blue = core, amber = correction surface, grey = operator context, red = blocked. The amber boundary marks the correction surface: the affected-party return path added on top of the paper.

The cryptography is real (nothing is mocked): the Ed25519 signing, JCS canonicalisation and Verifiable-Credential checks run in Python and are precomputed — this page renders the resulting verified trace.

▶ Where does NANDA sit? (the agent-standards landscape)

Read the agent stack as a ladder of questions. The standards answer them bottom-to-top — and stop at the same rung every time.

Rungs 1–4 (who is acting · on whose behalf · why · may it continue) are well covered — together they are the operator’s control surface. Rung 5 (who answers when a permitted action is wrong) is empty across the standards corpus, by charter — the governed party’s correction surface. NANDA sits at rung 1 (identity & discovery — its AgentFacts is a superset of the A2A Agent Card); this work’s correction surface (contestation + exit) is a protocol-layer attempt at the empty rung 5. The complete answer there is an institution (EU AI Act, product-liability law), not a credential — these map to the four layers in the sidebar.

Steward names are where each spec is maintained; every protocol below links to its primary source (verified). Interop (A2A), tools (MCP) and payments (AP2) ride on rungs 1–4 rather than extending the authority ladder.
